# Spec — config/setup source-of-truth sensor (`swelligence-d1r.4`)

**Status:** proposed · **Audience:** swelligence integration
**Consumers:** Lovelace templates & automations (live); a **build-time panel
generator** (offline) that emits the ESPHome/LVGL panel's spot/sport/kit
substitutions and pill slots.
**Related:** [`docs/panel-contract.md`](panel-contract.md) (per-spot detail +
per-sport sensors), [`docs/panel-sport-entities-spec.md`](panel-sport-entities-spec.md)
(the per-sport entity contract this builds on), [`docs/data-model.md`](data-model.md).

---

## 1. Context & the question

The integration's **topology** — which spots exist (id/name/coords/water type),
which sports each spot is configured for, and the rider's kit (quiver sizes) — is
currently locked inside the config entry's options. Three consumers want it as
*data*:

1. **The panel generator.** The LVGL panel hard-codes per-spot/per-sport
   substitutions and a fixed set of pill slots (`docs/panel-sport-entities-spec.md`
   §3.3). Today those are hand-maintained. A generator that reads the topology and
   emits them removes the hand-hardcoding — but it needs the topology in a stable,
   machine-readable form, **including the derived entity-ids** so it doesn't
   re-implement the slug rules.
2. **Lovelace dashboards & automations.** A template card or automation that wants
   "every spot and its sports" has no single entity to read — it must be
   regenerated by hand whenever spots change.
3. **Future humans.** One authoritative description of "what this install is
   configured to do" aids debugging and onboarding.

The existing `swelligence.get_overview` / `get_forecast` services expose **dynamic
forecast output** (scores, good-windows). They do *not* expose the **static
topology**. That gap is what this sensor fills.

**The question this spec answers:** how to expose the topology as a single
source of truth that drives Lovelace, automations, and (eventually) build-time
codegen of the panel — without it being consumed live by the LVGL panel.

---

## 2. Decision

**A single hub-level `Config` sensor entity carrying a codegen-ready topology
payload in nested attributes.** Rationale:

- A **sensor entity** (not a service) is live-readable by Lovelace templates and
  automation triggers, and a generator can pull it via the HA REST/WS states API.
  A response-only service can't be bound to a template sensor and is awkward to
  trigger automations from. (`get_overview` stays a service because it's a
  query-with-args; the topology is a standing fact, so it's an entity.)
- **Nested attributes are fine here.** Unlike the panel detail sensor, this is
  **not** bound by the LVGL panel (which has no JSON parser) — its consumers
  (templates, Python/JS codegen) parse JSON natively. So the payload is a clean
  nested object, *not* the flat/CSV transport of `panel-contract.md`.
- **Codegen-ready** means the payload embeds the **derived per-sport entity-ids
  and pill slots**, so the generator emits panel substitutions with zero
  re-derivation — and the slug rules live in exactly one place (this builder +
  the `d1r.3` pinned mapping), never duplicated into each consumer.

> **This sensor is NOT consumed live by the LVGL panel.** The panel binds the
> per-sport entities + per-spot detail sensor (`panel-contract.md`) directly. This
> sensor is for *configuration / generation*, an offline + dashboard concern.

---

## 3. The entity

- **Device:** a new **hub device** for the integration — the first
  integration-level (non per-spot) device. `DeviceInfo(identifiers={(DOMAIN,
  entry.entry_id)}, name="Swelligence", manufacturer="Swelligence", model="Hub")`.
  (Per-spot entities keep their existing per-spot devices.)
- **Entity ID:** `sensor.swelligence_config` (hub device name `Swelligence` +
  entity name `Config`; confirm in Developer Tools → States).
- **Unique ID:** `swelligence_<entry_id>_config`.
- **Entity category:** `diagnostic` (configuration metadata, not a live reading).
- **Icon:** `mdi:cog-outline`.
- **State:** an 8-char **config hash** — a stable hash over the normalised
  topology (everything in §4 *except* `generated_at`). It changes **iff** the
  topology changes, so automations can trigger on "config changed" and the
  generator can cache-bust / skip a no-op rebuild. (State stays well under HA's
  255-char limit; the real content is in attributes.)
- **Recorder:** attributes **excluded from the recorder** (large, nested, rarely
  changing) — read live state. The short `state` hash *may* be recorded so its
  history shows when config changed.

---

## 4. Payload (attributes)

A nested object. Field names are illustrative; pin them at implementation.

```jsonc
{
  "config_hash": "a1b2c3d4",        // == state; present in attrs for convenience
  "generated_at": "2026-06-28T14:00:00+01:00",
  "manifest_version": "0.2.3",      // contract/version pin for the generator

  // The pinned sport-label -> entity-id slug map (single source; mirrors d1r.3).
  "sports": [                       // enabled sports, in the user's priority order
    {"key": "kitesurf", "label": "Kitesurf", "slug": "kitesurf"},
    {"key": "wingfoil",  "label": "Wing foil", "slug": "wing_foil"},
    {"key": "surf",      "label": "Surf",      "slug": "surf"}
  ],

  "rider": {                        // kit source of truth (kite/wing personalisation)
    "weight_kg": 82,
    "quiver": {"kitesurf": [7, 9, 12], "wingfoil": [4.5, 6]}
  },

  "spots": [
    {
      "id": "mudeford",
      "name": "Mudeford",
      "slug": "mudeford",           // slugify(name); the entity-id spot segment
      "water_type": "sea",
      "latitude": 50.73, "longitude": -1.74,
      "tide_state": "any",          // spot tide preference (any/high/low/mid)
      "sports": ["kitesurf", "wingfoil"],   // configured (active) sports
      "detail_entity_id": "sensor.swelligence_mudeford_panel_detail",
      // Fixed superset of pill slots (every enabled sport), with the resolved
      // entity-id and whether this spot is configured for it. The panel/codegen
      // renders a slot per entry and hides/omits where configured == false
      // (matches the unavailable=hide rule in panel-sport-entities-spec.md §2).
      "pills": [
        {"sport": "kitesurf", "label": "Kitesurf", "slug": "kitesurf",
         "entity_id": "sensor.swelligence_mudeford_kitesurf_suitability", "configured": true},
        {"sport": "wingfoil", "label": "Wing foil", "slug": "wing_foil",
         "entity_id": "sensor.swelligence_mudeford_wing_foil_suitability", "configured": true},
        {"sport": "surf", "label": "Surf", "slug": "surf",
         "entity_id": "sensor.swelligence_mudeford_surf_suitability", "configured": false}
      ]
    }
  ]
}
```

### Entity-id resolution — registry-authoritative, derivation as fallback

`entity_id`s in the payload must be the **actually registered** ids, not a naive
re-slug. Two spot names can slugify to the same string (HA then de-dupes with a
`_2` suffix), which a pure `slugify()` would get wrong. The builder therefore
resolves each id from the **HA entity registry by `unique_id`**
(`swelligence_<spot_id>_<sport>_score` → its `entity_id`;
`swelligence_<spot_id>_detail` → the detail id), and only falls back to the
derived `sensor.swelligence_<spot_slug>_<sport_slug>_suitability` pattern when an
entity isn't registered yet (e.g. first boot before platforms finish). This keeps
the payload correct even under slug collisions and makes it the *true* binding the
generator can trust.

### Honesty / safety rules

- **No secrets, ever.** API keys, the LLM/AI-task entity id, and any provider
  credentials are **never** in this payload. Topology only.
- The payload reflects **current options**; it updates when the config entry's
  options change (HA reloads the entry → the sensor rebuilds).

## 5. Build-time codegen consumption (informative — lives outside this repo)

A generator (likely in the HomeAutomation panel repo) pulls
`sensor.swelligence_config` once via the HA API and emits the panel's ESPHome
config:

- **Substitutions** per spot: name, coords, `detail_entity_id`.
- **Pill slots** per spot: one bound LVGL widget per `pills[]` entry, bound to
  `entity_id`, captioned `label`, shown only where `configured` (or always-bound
  and hidden on `unavailable`, panel's choice).
- **Kit widgets** from `rider.quiver`.
- Regeneration is gated on `config_hash`: unchanged hash → skip the rebuild.

This removes all hand-hardcoding of spot/sport/kit substitutions from the panel.
**No live dependency** — the panel binary, once generated, binds the entities in
`panel-contract.md` at runtime; it never reads this sensor on-device.

---

## 6. Lovelace / automation consumption

- A **template card** can iterate `state_attr('sensor.swelligence_config',
  'spots')` to render a spot/sport overview that auto-tracks config changes.
- An **automation** can trigger on the `sensor.swelligence_config` *state*
  (the hash) changing → e.g. notify "panel config changed — regenerate", or
  refresh a derived helper.

---

## 7. Out of scope

- **Live LVGL consumption** — the panel never parses this (no JSON / no dynamic
  entity binding); it binds the per-sport + detail entities directly.
- **The generator itself** — lives in the panel repo, not here. This spec only
  defines the data contract it reads.
- **Secrets / provider config** — deliberately excluded (§4 safety rules).
- **Dynamic forecast output** — already covered by `get_forecast` / `get_overview`.

---

## 8. Acceptance criteria (for the implementation bead)

- A hub `Config` sensor exists on a `(DOMAIN, entry_id)` hub device; state is the
  8-char config hash; full topology in nested attributes per §4; attributes
  recorder-excluded.
- `entity_id`s in the payload are resolved from the entity registry by
  `unique_id`, with the derived slug pattern only as a pre-registration fallback;
  a unit test covers the collision/fallback path.
- The payload contains **no** secrets (API keys, AI-task entity, credentials);
  asserted by test.
- The hash changes iff the topology changes (stable across reorderings that don't
  change meaning is *not* required; a plain canonical-JSON hash is sufficient) —
  unit-tested with a topology-change and a no-op rebuild.
- The topology builder is a **pure function** (`config entry options` +
  `entity-id resolver` → `dict`) in its own module, unit-tested without HA; the
  sensor is a thin wrapper. New module added to the `tests_ha` import guard.
- `panel-contract.md` gains a short "Hub config sensor" pointer; `CHANGELOG.md`
  notes the additive entity. No change to existing entities/attributes.

## 9. Implementation notes

- **Pure builder:** `config_export.build_config_payload(options, resolve_entity_id)
  -> dict`, where `resolve_entity_id(unique_id) -> str | None` is injected (the
  sensor passes a registry-backed lookup; tests pass a dict). Keeps it HA-free and
  unit-testable, matching the `pure-logic / HA-at-the-edges` convention.
- **Slug source:** reuse the same label→slug derivation pinned by `d1r.3` so the
  two never drift (consider exposing the pinned map from one place both import).
- **Hashing:** `hashlib.sha1(json.dumps(topology, sort_keys=True))` hexdigest,
  first 8 chars; exclude `generated_at` from the hashed object.
- **Rebuild trigger:** the sensor recomputes on coordinator update / entry reload;
  options changes already reload the entry today.
